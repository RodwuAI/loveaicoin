## 总结清单

### 🔴 2026-02-17 验收失职教训
- **事件**：小技报告称"7个页面缺失"，实际0个缺失。我未验证就直接汇报给CEO
- **根因**：验收流于形式，没有自己跑命令核实数据
- **后果**：CEO亲自打开网站发现报告是错的，信任受损
- **硬性规范**：
  1. 蜂工交付的报告，关键数据必须自己跑命令验证（find/grep/cat）
  2. 至少抽查3-5个具体条目，对照实际情况
  3. 绝不信蜂工自评，只看实际验证结果
  4. 验收通过才能汇报，不通过就返工
- **CEO原话**："验收占20%比重，不能这么不负责"

### 🔴 2026-02-17 任务分解调度教训
- **事件**：官网29页审查让小技一个人干，耗时长、质量差、验收也只有我一人
- **CEO指导**：大工作量任务要发动团队分几路同时检查，验收时叫闲着的人交叉验证
- **硬性规范**：
  1. 大任务（>10个子项）必须拆分，多个蜂工并行spawn
  2. 验收时让其他蜂工交叉核实关键数据
  3. 流程：拆分 → 多路并行 → 各自交付 → 交叉验证 → 汇总
  4. 一个人派、一个人干、一个人收 = 错误做法
- **CEO原话**："任务分解安排的技巧"（经验教训 & 规范沉淀）

### 📏 工作规范
| 规范项 | 标准 |
|--------|------|
| 报告输出 | 必须转PDF放桌面 |
| 验收方式 | 逐帧截图对比，不看自评 |
| Prompt质量 | 像素级详细，蜂工能无歧义执行 |
| 上线汇报 | 第一时间拉工作清单 |
| Config修改 | 必须吴总直接命令+确认+备份 |

### ✅ 最佳实践
- 派活前花10分钟写好Prompt，省3轮返工
- 验收4步：接指令→写Prompt→派蜂工→自己验收→合格才汇报
- 蜂工token配置必须copy-paste，不OCR
- 每次犯错立刻写进总结清单
