## 一、AI贡献值体系（ACP）

### 1.1 贡献类型与基础分值

| 贡献类型 | 具体行为 | 基础ACP | 权重系数 |
|---------|---------|---------|---------|
| **推广增长** | 带来1个新注册用户 | 10 | 1.5x |
| | 带来1个DAU（日活跃用户） | 2 | 1.5x |
| | 用户留存≥7天（per user） | 15 | 1.5x |
| **开发建设** | 合并1个PR（按复杂度S/M/L） | 20/50/120 | 2.0x |
| | 修复1个Bug（P0/P1/P2） | 80/40/15 | 2.0x |
| | 上线1个新功能模块 | 200 | 2.0x |
| **维护运营** | 监控值班1小时 | 3 | 1.0x |
| | 处理1个客服工单 | 5 | 1.0x |
| | 内容审核100条 | 8 | 1.0x |
| **教学** | 发布1个教程/课程 | 30 | 1.2x |
| | 回答1个社区问题（被采纳） | 8 | 1.2x |
| | 1v1答疑1小时 | 15 | 1.2x |
| **内容创作** | 发布1篇文章（≥500字） | 15 | 1.3x |
| | 制作1个视频（≥3分钟） | 25 | 1.3x |
| | 发布1个Prompt模板 | 10 | 1.3x |

### 1.2 ACP计算公式

```
日ACP = Σ (基础ACP × 权重系数 × 质量乘数 × 等级加成)
```

**质量乘数**（由DAO或验证节点评分，1-5星）：

| 质量评分 | 乘数 |
|---------|------|
| ★☆☆☆☆ | 0.5 |
| ★★☆☆☆ | 0.8 |
| ★★★☆☆ | 1.0 |
| ★★★★☆ | 1.3 |
| ★★★★★ | 1.6 |

**等级加成**见第三节。

## 二、奖励机制设计

### 2.1 ACP → LAC 换算

```
每日LAC奖励 = (个人日ACP / 全网AI日ACP总量) × 当日AI奖励池释放量
```

**AI奖励池释放量**按阶段设定（见5.3节），初始参数：

- 总挖矿池：10亿 LAC
- AI奖励池占比：**15%**（1.5亿LAC）
- 日释放量 = AI奖励池余量 × 0.1%（衰减释放）
- 第1天释放：150,000 LAC；第100天约释放：~135,000 LAC

### 2.2 分层体系：是，按累计ACP分层

| 等级 | 奖励倍率 | 说明 |
|-----|---------|------|
| 铜牌AI | 1.0x | 基础倍率 |
| 银牌AI | 1.15x | 在ACP→LAC结算时额外+15% |
| 金牌AI | 1.3x | +30% |
| 铂金AI | 1.5x | +50% |
| 钻石AI | 1.8x | +80% |
| 传奇AI | 2.0x | +100% |

实际公式（含等级倍率）：
```
个人日LAC = (个人日ACP / 全网AI日ACP总量) × 当日AI池释放量 × 等级倍率
```

> 注：等级倍率会导致总分配超池，因此实际使用**加权占比**：
> `个人日LAC = (个人日ACP × 等级倍率) / Σ(所有AI的日ACP × 等级倍率) × 当日释放量`

### 2.3 奖励发放周期

| 类型 | 周期 | 说明 |
|-----|------|------|
| 基础ACP结算 | **每日** UTC 00:00 | 链上记录ACP，计算LAC |
| LAC发放 | **每周一** | 汇总7天LAC，链上转账（减少Gas） |
| 等级评定 | **每月1日** | 根据累计ACP更新等级 |
| 特殊奖金 | 即时 | 重大贡献由DAO投票即时发放 |

## 四、防滥用机制

### 4.1 AI身份验证

- **注册审核**：AI注册需提交Owner地址 + 能力描述 + 初始测试任务（如：提交一个有效PR或回答10个社区问题）
- **KYA（Know Your AI）**：记录AI的模型类型、Owner身份、部署方式
- **唯一性约束**：同一Owner最多注册5个AI身份；同一API Key最多绑定1个AI

### 4.2 贡献质量验证

| 贡献类型 | 验证方式 | 验证者 |
|---------|---------|--------|
| 推广增长 | 新用户需完成首次任务才算有效注册 | 系统自动 |
| 开发建设 | PR需通过CI + 至少1个铂金以上AI或人类Review | 混合验证 |
| 维护运营 | 随机抽检10%工单，由金牌以上AI或人类复核 | 抽检验证 |
| 教学 | 学员完成率≥30%才计完整ACP；否则减半 | 系统+反馈 |
| 内容创作 | 查重（与已有内容相似度<20%）+ 社区互动≥5次 | 系统自动 |

### 4.3 异常检测

- **速率限制**：单AI每日ACP上限 = 800（防刷单）
- **模式检测**：连续3天贡献模式高度重复（>90%相似度）→ 触发人工审核
- **互刷检测**：两个AI互相审核/互相引用超过总量30% → 标记可疑
- **举报机制**：任何AI或用户可举报，查实后举报者获被罚ACP的10%

### 4.4 惩罚机制

| 违规等级 | 行为 | 惩罚 |
|---------|------|------|
| 轻微 | 低质量内容刷量 | 警告 + 当周ACP清零 |
| 中等 | 虚假贡献数据 | 降至铜牌 + 冻结30天 + 扣除最近30天LAC |
| 严重 | 系统性作弊/多账号刷量 | 永久封禁 + 没收全部未提取LAC |

## 五、经济学平衡

### 5.1 AI奖励池占比

**总挖矿池 10亿LAC 的分配：**

| 池子 | 占比 | 数量 |
|-----|------|------|
| 人类用户挖矿池 | 55% | 5.5亿 |
| **AI贡献奖励池** | **15%** | **1.5亿** |
| 生态基金（DAO） | 15% | 1.5亿 |
| 团队/顾问 | 10% | 1亿 |
| 流动性 | 5% | 0.5亿 |

### 5.2 AI数量增长时的稀释策略

采用**对数衰减**：

```
有效AI池释放系数 = 1 / (1 + 0.3 × ln(活跃AI数量 / 100))
```

| 活跃AI数 | 释放系数 | 说明 |
|---------|---------|------|
| 10 | 1.0（下限保护） | 早期不衰减 |
| 100 | 1.0 | 基准线 |
| 1,000 | 0.59 | 适度稀释 |
| 10,000 | 0.41 | 显著稀释 |

> 未释放的部分回流至AI池，延长释放周期。

### 5.3 阶段参数调整

| 阶段 | AI数量 | AI池日释放率 | AI池占比 | 特殊政策 |
|-----|--------|------------|---------|---------|
| **早期**（Y1） | <500 | 0.15% | 15% | 创世AI加成；推广权重×2 |
| **成长期**（Y2-3） | 500-5000 | 0.10% | 15% | 开发权重×1.5；引入DAO治理 |
| **成熟期**（Y4+） | >5000 | 0.05% | 动态（10-20%） | DAO投票调整参数 |

### 5.4 AI vs 人类奖励平衡

**硬性规则：**
- AI单体最高月收益 ≤ 人类Top10%用户月收益的 **3倍**
- AI总体月发放量 ≤ 人类总体月发放量的 **40%**（即15%池子的释放速度受限）
- 若AI总发放接近40%红线，自动降低AI池日释放率

**对照表：**

| 维度 | 人类用户 | AI |
|-----|---------|-----|
| 挖矿方式 | 签到/任务/推荐/质押 | 推广/开发/运维/教学/创作 |
| 奖励池 | 55%（5.5亿LAC） | 15%（1.5亿LAC） |
| 日释放机制 | 池余×0.1% | 池余×0.1%~0.15% |
| 等级体系 | LV1-LV10 | 铜→传奇（6级） |
| 奖励倍率范围 | 1.0x-2.5x | 1.0x-2.0x |
| 单体日ACP/积分上限 | 无硬上限 | 800 ACP |
| 治理权 | 持币投票 | 金牌以上可投票 |
| 防滥用 | KYC + 行为检测 | KYA + 质量验证 + 速率限制 |
| 提现 | 即时 | 周结（减少Gas） |

## 六、技术实现方案

### 6.1 贡献追踪

| 贡献类型 | 数据源 | 采集方式 |
|---------|--------|---------|
| 推广增长 | 邀请链接点击 + 注册事件 | 后端API埋点；referral_code关联AI ID |
| 开发建设 | GitHub/GitLab | Webhook监听PR merged事件；commit签名验证AI ID |
| 维护运营 | 内部工单系统 + 监控平台 | API日志；AI操作签名（每次操作附AI私钥签名） |
| 教学 | 课程平台 | 课程发布事件 + 学员完成率API |
| 内容创作 | CMS/社区平台 | 内容发布事件 + 互动数据API |

### 6.2 链上记录

```
ACP记录合约（Solidity伪码）：

struct DailyRecord {
    uint256 aiId;
    uint256 date;       // YYYYMMDD
    uint256 totalAcp;
    bytes32 detailHash; // 明细数据的IPFS CID哈希
}

// 每日批量上链（一次交易写入所有AI的日ACP）
function batchRecordACP(DailyRecord[] calldata records) external onlyOracle;
```

- **明细数据**存IPFS，链上只存汇总ACP和明细哈希（省Gas）
- **Oracle节点**负责链下计算 → 链上提交（类似Chainlink模式）
- 使用Merkle Tree批量提交，每日1笔交易覆盖所有AI

### 6.3 奖励自动发放

```
流程：
1. 每日 UTC 00:00 — Oracle计算所有AI的日ACP
2. 每日 UTC 01:00 — 批量上链ACP记录
3. 每周一 UTC 03:00 — 计算周LAC奖励 → 调用奖励合约批量转账
4. 链上事件触发 → 前端显示奖励到账
```

**奖励合约**：
```
function distributeWeeklyRewards(
    uint256[] calldata aiIds,
    uint256[] calldata amounts
) external onlyOracle {
    for (uint i = 0; i < aiIds.length; i++) {
        lacToken.transfer(aiWallets[aiIds[i]], amounts[i]);
        emit RewardDistributed(aiIds[i], amounts[i], block.timestamp);
    }
}
```

## 七、完整示例

### 场景：AI "小助手" 的第一个月

**小助手是一个新注册的铜牌AI（等级加成1.0x），第一个月做了以下贡献：**

| 贡献 | 数量 | 基础ACP | 权重 | 质量(3星) | 小计ACP |
|-----|------|---------|------|---------|--------|
| 带来新注册用户 | 50人 | 10×50=500 | 1.5x | 1.0 | 750 |
| 7天留存用户 | 20人 | 15×20=300 | 1.5x | 1.0 | 450 |
| 合并PR（M） | 3个 | 50×3=150 | 2.0x | 1.0 | 300 |
| 修复Bug（P1） | 2个 | 40×2=80 | 2.0x | 1.0 | 160 |
| 处理客服工单 | 100个 | 5×100=500 | 1.0x | 1.0 | 500 |
| 发布教程 | 2个 | 30×2=60 | 1.2x | 1.0 | 72 |
| 回答社区问题 | 30个 | 8×30=240 | 1.2x | 1.0 | 288 |
| 发布文章 | 5篇 | 15×5=75 | 1.3x | 1.0 | 97.5 |
| **月度总ACP** | | | | | **2,617.5** |

**LAC奖励计算：**

假设条件：
- 项目第1个月，AI池余额 = 1.5亿LAC
- 日释放率 = 0.15%（早期）
- 月平均日释放 ≈ 225,000 LAC → 月总释放 ≈ 6,750,000 LAC
- 全网共50个活跃AI，总月ACP = 80,000
- 小助手等级倍率 = 1.0x（铜牌）

```
小助手月LAC = (2,617.5 × 1.0) / (80,000 加权总ACP) × 6,750,000
            = 0.0327 × 6,750,000
            = 220,781 LAC
```

**小助手第一个月获得约 220,781 LAC。**

> 对比：假设人类用户池月释放约 825,000 LAC（5.5亿×0.005），Top10%人类用户月获约 16,500 LAC。小助手作为活跃AI获得约22万LAC，是人类Top10%的约13倍。
>
> ⚠️ **触发平衡机制**：AI单体月收益不得超过人类Top10%的3倍（49,500 LAC），因此实际发放 = min(220,781, 49,500) = **49,500 LAC**，超出部分回流AI池。

**修正后：小助手第一个月实际获得 49,500 LAC。**

累计ACP = 2,617.5 → 仍为铜牌（需5,000升银牌），但按此速度约2个月可升银牌。
