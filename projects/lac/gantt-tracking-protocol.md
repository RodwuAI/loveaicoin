# 甘特图跟踪管理协议 V1.0

> 甘特图 = 作战地图。每两小时迭代一次。——吴总指令 2026-02-22

---

## 一、为什么甘特图是核心中的核心

传统项目管理中，甘特图更新频率通常是每周一次。但LAC项目是11天冲刺，等同于一个超短Sprint——**一周更新一次等于项目结束才更新**。

我们的实际情况：
- 11天冲刺 = 每天等于传统项目的一周
- 10个蜂工并行 = 高并发协作
- 吴总是CEO不是PM = 甘特图是他唯一的全局视野
- 任务完成速度极快 = 2小时内状态可能变化很大

所以**每2小时更新一次甘特图**，在我们的场景下完全合理，相当于传统项目的"每日站会"。

## 二、当前问题诊断

| 问题 | 表现 | 根因 |
|------|------|------|
| 甘特图滞后 | V1还写着"创业者1/5"，实际已经3/5 | 没有定时更新机制 |
| 版本断裂 | V1→V2是重做不是迭代 | 没有迭代协议 |
| 缺少关键指标 | 只有"完成/执行中/待启动" | 没有EVM挣值分析 |
| 无阻塞预警 | CEO任务阻塞看不出来 | 没有依赖关系可视化 |
| 无关键路径 | 不知道哪条链路决定上线日期 | 没有CPM分析 |

## 三、优化方案

### 3.1 自动化2小时迭代机制

**用Cron Job每2小时触发甘特图更新：**

1. 检查所有活跃子会话的完成状态
2. 读取 `full-project-plan-v3.csv` 当前状态
3. 对比实际进展，更新CSV中的状态列
4. 重新生成HTML甘特图
5. 生成PDF放桌面
6. 如有重大变化（关键路径任务完成/阻塞），主动通知吴总

### 3.2 引入EVM挣值分析

为甘特图增加三个核心EVM指标：

| 指标 | 公式 | 含义 | 我们怎么算 |
|------|------|------|----------|
| **PV (计划值)** | 截至今天应完成的任务数 | 计划进度 | 按CSV截止日期统计 |
| **EV (挣值)** | 实际完成的任务数 | 实际产出 | 按CSV状态=✅统计 |
| **SPI (进度绩效)** | EV / PV | >1=超前, <1=滞后 | 自动计算 |

**当前LAC项目EVM快照：**
- PV (Day3应完成): ~25个任务（T001-T022 + 部分T023-T033）
- EV (实际完成): 40个任务（✅已完成）
- SPI = 40/25 = **1.6** → 进度超前60%！
- 但CEO任务SPI = 1/8 = **0.125** → CEO任务严重滞后

### 3.4 依赖关系预警

每次更新甘特图时，自动检查：
1. **阻塞链分析**：哪些待启动任务的前置依赖已完成？→ 可以启动
2. **CEO瓶颈检测**：CEO任务是否在关键路径上？延迟多少天？
3. **空闲蜂工检测**：哪个蜂工没有活跃任务？→ 立即安排

### 3.5 甘特图版本迭代规则

| 规则 | 说明 |
|------|------|
| **永远在最新版上改** | 不新建文件，直接修改 `gantt-chart-v3.html` → 原地更新 |
| **CSV是唯一数据源** | `full-project-plan-v3.csv` → 手动/自动更新状态列 |
| **版本号只在重大结构变化时升** | 日常更新不升版本号，只改"更新时间" |
| **每次更新写变更日志** | 在HTML底部记录"本次更新：T018完成、T055修复中" |

### 每次更新流程（2小时一次）

```
6. 生成PDF覆盖桌面旧版
7. 如有重大变化 → 通知吴总
8. 记录变更日志
```

### Cron实现
- 用OpenClaw cron每2小时触发
- payload: "检查LAC项目所有任务进度，更新甘特图CSV和HTML，生成PDF放桌面"
- sessionTarget: isolated（不污染主会话）

## 五、参考方法论

| 方法 | 我们的应用 |
|------|----------|
| **CPM关键路径法** | 识别CEO任务→预热→内测→上线的关键依赖链 |
| **EVM挣值管理** | SPI监控整体进度健康度，分维度（蜂工/CEO） |
| **看板限制WIP** | 每个蜂工同时只做1个任务，完成一个再接下一个 |
| **Daily Standup** | 2小时微站会 = 甘特图更新 |
| **风险登记册** | 阻塞项面板 = 实时风险看板 |
| **燃尽图思维** | 剩余任务数随时间递减，偏离基线=预警 |
